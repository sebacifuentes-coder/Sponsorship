- (event) Modo guiado (coaching con recomendaciones). Entregables: spine + explicador visual. Inputs: PRD v2.4 + project-context.md
- (decision) AD-1 [ADOPTED]: Arquitectura de DOBLE PLANO como monolito modular. Plano ligero (datos publicos club+hincha -> motor oportunidades + generacion IA -> web app, valor <10min, sin PII ni integracion enterprise) + Plano profundo (clean room first-party, conectores Salesforce/Adobe, motor de valor medible, progresivo en segundo plano). Binds: separacion de planos; el ligero nunca depende del profundo. Prevents: que el TTFV se rompa por dependencias de integracion
- (version) Verificado en web (jun 2026): Next.js 16.2.x estable (App Router + Turbopack por defecto). Consenso 2026 MVP IA: Next.js + Supabase + Vercel AI SDK
- (decision) AD-2 [ADOPTED]: Stack = Next.js (App Router, v16.x) + Supabase (Postgres+auth+storage) + Vercel AI SDK + Tailwind/shadcn. LLM via provider (Azure OpenAI/Google) a traves del AI SDK. TypeScript unico lenguaje. Worker Python opcional a futuro para el plano profundo. Binds: lenguaje, framework, BD, capa IA. Prevents: divergencia de stack y friccion en handoff al dev
- (decision) AD-3 [ADOPTED]: Frontera de datos. Dos dominios separados: Almacen de Inteligencia Publica (datos publicos club+hincha, SIN PII, alimenta plano ligero) y Clean Room (first-party PII, aislado). Regla: la PII NUNCA cruza hacia la marca ni al plano ligero; solo cruzan segmentos agregados/anonimizados. Binds: ubicacion y flujo de datos sensibles. Prevents: fuga de PII / incumplimiento GDPR / acoplar datos sensibles al plano ligero
- (decision) AD-4 [ADOPTED]: Motor de valor medible. Metodologia de atribucion como artefacto VERSIONADO y firmado; cada evento de Valor medible referencia la version vigente y queda en registro append-only auditable; certificacion SEPARADA del cobro (rol auditoria/disputa independiente). Regla: no hay evento de valor sin version de metodologia referenciada; eventos inmutables, se auditan. Binds: como se mide/certifica/cobra el valor. Prevents: juez-y-parte, sobre-atribucion, disputas sin traza
- (decision) AD-5 [ADOPTED]: Conectores como adaptadores. Cada sistema externo (Salesforce, Adobe, proveedor LLM, fuentes de datos publicos) detras de un adaptador con contrato interno estable; el nucleo nunca llama al SDK del proveedor directo. Binds: como se integra lo externo. Prevents: acoplamiento a un vendor, bloqueo de la integracion progresiva del plano profundo
- (decision) AD-6 [ADOPTED]: Despliegue y entornos. Local (Next.js + Supabase local) -> preview automatico por cambio en GitHub -> produccion (Vercel app + Supabase gestionado). Secretos en variables de entorno, nunca en el repo. Binds: entornos y donde corre cada cosa. Prevents: secretos filtrados, drift entre entornos
- (decision) AD-7 [ADOPTED]: Ciclo de vida de la activacion. Estados borrador->aprobada->publicada, cada transicion con autor/fecha. El nucleo/dominio es UNICO dueno del estado; los conectores solo ejecutan la publicacion, no mutan estado. Binds: quien muta el estado de la Activacion. Prevents: dos duenos del estado, conectores cambiando estado de forma inconsistente
- (decision) AD-8 [ADOPTED]: Contrato de segmento agregado. Clean Room expone catalogo cerrado de segmentos pre-agregados (no query libre); tipo SegmentoAgregado con k-minimo, supresion de celdas pequenas, lista blanca de atributos, sin IDs por-fan; puerto valida en runtime. Prevents: re-identificacion por segmentos pequenos o consultas diferenciales
- (decision) AD-9 [ADOPTED]: Identidad/roles/tenancy. RBAC de 3 actores + RLS de Supabase como frontera tecnica de AD-3 (Marca nunca lee PII del clean room); multi-tenant por Propiedad con aislamiento reforzado del clean room. Prevents: fuga de PII a la marca, una Propiedad viendo datos de otra
- (event) Gate adversarial corrido: 4 huecos de divergencia + 5 dimensiones en silencio detectados y resueltos (AD-7 generalizado, AD-8, AD-9, degradacion en AD-1, idempotencia en AD-5, observabilidad/TTFV en convenciones). Spine finalized (status: final)
